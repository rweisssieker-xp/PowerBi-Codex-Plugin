## Priority Expert-Agent Skills

The first planned skills must prove technical trust, not only marketing breadth:

- `powerbi-ai-model-graph-validator`: checks relationship topology, ambiguous paths, fact-to-fact chains, role-playing dates, inactive relationship strategy, and unsupported composite-model patterns.
- `powerbi-ai-native-source-router`: chooses correct native Power Query connector patterns for Excel, CSV, Folder, SharePoint, SQL, Fabric, APIs, and SaaS sources.
- `powerbi-ai-visual-binding-validator`: checks PBIR visual bindings against actual model fields and measures.
- `powerbi-ai-powerbi-desktop-smoke-tester`: opens generated PBIP in target Power BI Desktop, checks process state, Frown snapshots, and load outcome.
- `powerbi-ai-frown-to-fix-triage`: classifies Power BI Desktop Frown errors and maps them to likely PBIP/TMDL/PBIR/M/DAX fixes.

## Skill Input and Output Standard

Every expert-agent skill must document:

- Input: business request, process scope, source metadata, existing PBIP/TMDL/PBIR, KPI contract, or error log.
- Output: concrete artifact such as model design, DAX catalog, M query, report layout, validation report, release evidence, or fix plan.
- Guardrails: assumptions, missing evidence, confidence, required owner approval, and quality gate.
- Handoff: next skill or artifact that consumes the output.

## Acceptance Rule

A skill cluster counts as expert replacement only if it produces both a business artifact and a technical validation artifact. For example, a DAX Expert Agent must produce measures and tests; a Designer Agent must produce pages and valid visual bindings; a Power Query Expert Agent must produce M code and a load/refresh contract.
